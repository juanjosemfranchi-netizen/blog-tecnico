Post-Mortem: Optimización de Pipelines de Automatización (IA)

Autor: Juan José M. Franchi | Rol: Systems Analyst

1. Introducción

Este informe detalla la evolución técnica de un sistema de automatización para la carga de productos en e-commerce. El objetivo fue mitigar la ineficiencia operativa mediante la integración de LLMs, buscando una estructura de salida predecible y profesional.

2. Análisis de Fallas: ¿Qué observamos en producción?

Tras las primeras ejecuciones del pipeline, detectamos errores críticos que bloqueaban la carga en el backend. Los incidentes más recurrentes fueron:

  • JSON inválido: El modelo incluía caracteres especiales o texto narrativo fuera del bloque de código, rompiendo el proceso de parsing.
  • Omisión de campos obligatorios: En productos complejos, la IA priorizaba la creatividad, dejando secciones técnicas vacías.

Ejemplo: El backend rechazaba automáticamente la carga al no poder procesar el JSON recibido o al detectar campos obligatorios vacíos.

Causa Raíz: Ausencia de un contrato de interfaz rígido; el modelo actuaba como redactor creativo y no como servicio de datos.

3. Estrategia de Refactorización y Validación

Implementamos una estrategia de "Zero-Shot con esquema rígido" para forzar una respuesta en formato JSON validado.

Acción Técnica: Refactorización del prompt incluyendo roles definidos, temperatura (0.2) y esquemas obligatorios.
Evidencia: Commit de Refactorización de Prompt.

4. Métricas de Desempeño

La eficacia se validó comparando el comportamiento del pipeline procesando 100 ejecuciones (cada una correspondiente a un ítem del catálogo):

Antes: ~35% fallas de parseo (sobre las primeras 50 ejecuciones)
Después: <1% fallas de validación (en las últimas 100 ejecuciones)

5. Reglas de Oro Aprendidas

  • Contrato de Interfaz: Todo output de IA debe tratarse como una API: si no cumple el esquema, se rechaza.
  • Gatekeeping: Ninguna salida llega a producción sin pasar por un script de validación (tests.py).
  • Gestión de Cambios: Cualquier ajuste en el prompt es una modificación de contrato y debe versionarse (ver implementación de tests.py).

6. Reflexión sobre el Feedback

Feedback recibido: El proyecto carecía de evidencia de errores concretos y una narrativa que justificara la robustez técnica.

Autocrítica: Me di cuenta de que estaba describiendo el "qué" funcional, pero no el "por qué" técnico con evidencia suficiente.

Cambio aplicado: Integré el análisis cuantitativo de fallas, la suite de validación automatizada y adopté Conventional Commits para mejorar la trazabilidad. Esta evolución permitió elevar el estándar de mis entregables, pasando de un enfoque funcional a uno auditable y profesional.
