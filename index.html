Post-Mortem Técnico: Optimización y Escalabilidad de Pipeline de IA

Autor: Juan José M. Franchi | Estatus: Proyecto en Producción

1. Contexto y Problema

El proyecto nació bajo una premisa simple: la operación manual de carga de productos para el e-commerce estaba limitando el crecimiento. El proceso tradicional, que involucraba redactar manualmente descripciones, títulos y tags, se realizaba en silos, sin un estándar de calidad. Los datos de entrada eran heterogéneos, variando desde descripciones ricas hasta simples "códigos de producto" sin contexto. Esto no solo consumía 15 minutos por SKU, sino que introducía errores humanos constantes.

2. Arquitectura del Flujo y Ajustes Técnicos

La implementación consistió en un flujo de tres capas:

  • Capa de Ingesta: Un formulario centralizado que normaliza la entrada de datos mínimos.
  • Capa de Inteligencia: API de OpenAI (GPT-4o). Aquí enfrentamos el mayor desafío: la falta de determinismo. Para solucionar esto, implementamos una configuración técnica específica: temperature: 0.2 (buscando bajo riesgo y alta precisión) y forzamos un contrato mediante un System Prompt que define una salida en formato JSON con campos: titulo_seo, descripcion_comercial y caracteristicas_tecnicas.
  • Capa de Publicación: Integración mediante Webhooks hacia la plataforma destino.

3. Análisis Post-Mortem y Métricas de Impacto

Tras la primera iteración, los resultados fueron mixtos. Si bien la automatización estaba activa, la calidad del texto no convertía en ventas. El post-mortem reveló que nuestro prompt trataba al modelo como un "escritor creativo" cuando el e-commerce requiere un "especialista en ventas".

Tiempo Manual: 15 min/SKU
Tiempo Automatizado: 30 seg/SKU
Eficiencia: 96% de ahorro

La refactorización se llevó a cabo utilizando un flujo de trabajo profesional mediante ramas en Git, asegurando que cada cambio en el prompt fuera revisado y testeado antes de pasar a producción. Ver Commit de Refactorización.

4. Validación mediante Integración Continua

Un error común en proyectos de IA es confiar ciegamente en el modelo. Para mitigar esto, desarrollamos una suite de validación de código. El archivo tests.py no es solo un script; es un contrato de calidad que verifica que el JSON generado no esté roto o mal formado. Si el test falla, el pipeline se detiene. Ver registro de pruebas automáticas.

5. Reflexión sobre el Feedback y Crecimiento Profesional

La recepción de feedback crítico durante este proceso fue el punto de inflexión de mi desarrollo como Systems Analyst. Inicialmente, mi percepción estaba sesgada hacia la "funcionalidad" (si el código corría, estaba bien). El feedback me obligó a cuestionar la "calidad de la arquitectura".

¿Cómo impactó esto? Cambié la metodología. Pasé de realizar commits directos a adoptar ramas de trabajo y realizar Pull Requests auto-revisados. Entendí que el código es comunicación: si mi documentación (o mi post-mortem) no explica el 'por qué' y el 'cómo', el código es inútil para el equipo. Esta evolución me ha permitido elevar los estándares de mis entregables, integrando la trazabilidad como parte de mi identidad como desarrollador.
